## 1. 行业通用能力

| 能力 | 行业通用做法 | 本平台设计 |
|---|---|---|
| 多业务接入 | 业务线通过标准事件接入，平台不理解业务私有表。 | `BusinessAdapter` + `SourceEvent`。 |
| 清分 | 单笔订单按规则拆分金额、成本、佣金、分润。 | `ClearingResult` + `ClearingResultItem`。 |
| 清算 | 周期汇总、账期成熟、冻结/解冻、轧差、应收应付确认。 | `ClearingBatch` + `PendingSettlementItem`。 |
| 结算 | 生成结算批次、结算单、明细和凭证，驱动入账。 | `SettlementBatch` + `SettlementBill`。 |
| 账务 | 余额和流水由账户账务平台承接。 | `AccountingOrder` 调用 AccountingFacade。 |
| 出款 | 结算完成后商户可提现，出款平台处理。 | 边界衔接，不在一期核心。 |
| 对账 | 订单、清分、清算、结算、账务、出款多层对账。 | `Reconciliation` 上下文。 |
| 差错 | 漏清分、漏结算、重复入账、金额不一致可处理。 | P1 建设差错工作台。 |

## 2. 设计吸收的行业经验

- 美团旅行结算平台的核心经验是结算平台化，支持多业务线接入，把账单、付款、发票等能力平台化，新业务低成本接入。
- 抖音/快手生活服务交易类平台的资金链路通常区分在途/冻结资金和可提现资金，达到履约和账期后再结算。
- 支付宝直付通、腾讯云商付等平台型资金产品强调支付、分账、结算、对账清算和商户进件的协同。
- 合规上，平台不能自行挪用或占用客户备付金，清结算平台需要和支付机构、账户账务、出款平台保持边界。

## 3. 本平台取舍

| 设计点 | 取舍 |
|---|---|
| 一期是否做完整规则引擎 | 不做，先做规则版本和快照，复杂 DSL 后置。 |
| 一期是否自动出款 | 不做，结算入账后由商户提现。 |
| 一期是否做完整对账工作台 | 不做，先做最小对账键和结算单 vs 账务流水校验。 |
| 一期是否兼容旧表 | 是，通过防腐层兼容旧数据，不把旧表作为新核心。 |
| 是否支持按摩平台 | 总体模型支持，P2 通过 adapter 接入。 |
