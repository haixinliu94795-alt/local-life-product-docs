# 事务边界

## 1. 本章结论

P0 事务边界以资金事实正确为第一优先级。结算确认主事务只包含必须保持一致的业务状态：头寸锁定、结算批次、结算单、结算明细、账务入账单。操作日志不作为资金事实来源，不允许因为日志写入失败导致资金主事务回滚。

## 2. 结算确认主事务

统一结算确认方法：

```java
SettlementBillNo confirmMerchantSettlement(ConfirmMerchantSettlementCommand command)
```

主事务内步骤：

| 步骤 | 动作 | 是否主事务内 |
|---|---|---:|
| 1 | 校验幂等键和 request_hash | 是 |
| 2 | 校验 positionNos 数量不超过 500 | 是 |
| 3 | 查询并锁定 `ELIGIBLE` 头寸 | 是 |
| 4 | 校验同一商户、同一业务场景、同一结算主体 | 是 |
| 5 | 创建 `SettlementBatch` | 是 |
| 6 | 创建 `SettlementBill` | 是 |
| 7 | 创建 `SettlementBillItem` | 是 |
| 8 | 将头寸推进到 `LOCKED` | 是 |
| 9 | 创建 `AccountingPostingOrder` | 是 |
| 10 | 主事务提交 | 是 |
| 11 | 写 `OperationLog` | 否，afterCommit / REQUIRES_NEW / 异步 |
| 12 | 调用账务平台 | 可在主事务后执行，按实现策略选择同步或异步 |

## 3. 操作日志事务策略

### 3.1 设计原则

`ccs_operation_log` 是运营审计和诊断辅助数据，不是资金事实唯一来源。资金事实以 `SettlementPosition / SettlementBill / AccountingPostingOrder` 为准。

### 3.2 推荐实现

```java
@Transactional
public SettlementBillNo confirmMerchantSettlement(command) {
    // 1. 幂等、校验、锁定头寸
    // 2. 创建 batch/bill/items/posting order
    // 3. 注册 afterCommit 回调
    TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronizationAdapter() {
        @Override
        public void afterCommit() {
            operationLogWriter.writeRequiresNew(...);
        }
    });
    return billNo;
}
```

### 3.3 失败策略

| 场景 | 处理 |
|---|---|
| 主事务失败 | 不写操作日志或写失败诊断日志，不生成资金事实。 |
| 主事务成功，操作日志失败 | 不回滚主事务，打 error log 和 metric。 |
| 操作日志重复写入 | 使用 `operation_no` 幂等或允许查询侧去重。 |
| 审计强依赖场景 | 后续可通过消息/任务补偿操作日志，但 P0 不阻塞资金主链路。 |

## 4. 账务入账事务边界

账务入账跨平台调用不能和本地数据库事务组成强事务。P0 采用本地状态机 + 账务幂等 + UNKNOWN 查询补偿。

| 阶段 | 本地动作 | 外部动作 | 一致性策略 |
|---|---|---|---|
| 创建入账单 | `AccountingPostingOrder=INIT` | 无 | 本地事务 |
| 发起入账 | `REQUESTING` | 调用账务平台 | 账务幂等键 |
| 成功 | `SUCCESS`，结算单/头寸 `ACCOUNTED` | 账务流水生成 | 状态联动 |
| 失败 | `FAILED`，结算单/头寸 `ACCOUNTING_FAILED` | 无成功入账 | 可重试 |
| 未知 | `UNKNOWN` | 不确定 | 先查询，不直接重入账 |

## 5. 禁止项

1. 禁止把操作日志写入失败作为结算主事务回滚原因。
2. 禁止在 UNKNOWN 状态下直接重复调用入账。
3. 禁止结算单已 `ACCOUNTED` 后重新生成入账单。
4. 禁止将外部账务调用包进本地长事务中持锁等待。
