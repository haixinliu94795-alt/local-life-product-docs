# 当前规范

## 1. 文档定位

本文档是清结算平台 V005 SDD 开发准入版，面向研发和 Codex 执行。它不是产品 PRD，也不是旧系统整改说明。

## 2. 核心约束

| 约束 | 说明 |
|---|---|
| 领域优先 | 底层模型按本地生活/电商成熟清结算平台的标准领域语言建模。 |
| 不被当前产品带偏 | 当前原型只作为一期场景输入，不反向定义清结算平台底层状态和表模型。 |
| 不被旧系统带偏 | 旧表和旧字段只在参考输入中保留，不作为新平台核心模型。 |
| 正向闭环优先 | P0 只实现履约/核销完成后的正向清分、清算、结算、账务入账。 |
| 退款后置 | 退款、撤销、冲正属于逆向清分/调账能力，P0 不设计字段、不设计事件、不设计状态流转。 |
| BFF 后置 | 上层商户端展示口径不进入底层设计；清结算平台只提供标准领域状态和查询能力。 |
| 账务标准契约 | 清结算平台定义标准账务入账端口，账户账务平台按契约配合。 |
| 不做冻结止付 | 冻结、止付、风控锁定均不属于 P0。 |

## 3. P0 主线

```text
履约/核销完成事件
  -> 清分结果
  -> 结算头寸 Settlement Position
  -> 账期成熟后进入可结算状态
  -> 运营确认单条/批量结算
  -> 结算批次、结算单、结算明细
  -> 账务入账单
  -> 调用账户账务平台入账
  -> 商户账户余额增加
```

# 通用语言

| 术语 | 英文 | 定义 |
|---|---|---|
| 来源事件 | Source Event | 外部业务平台推送给清结算平台的标准业务事实。P0 只接履约/核销完成。 |
| 清分 | Clearing | 单笔业务维度计算资金归属，输出商户应结、平台应收等清分明细。 |
| 清分结果 | Clearing Result | 一笔业务事实对应的资金归属计算结果。 |
| 清分明细 | Clearing Result Item | 清分结果中面向一个参与方和一个金额类型的明细。 |
| 在途资金 | In-transit Amount | 已形成交易事实但未达到结算条件或未完成结算入账的资金。 |
| 清算 | Settlement Preparation / Clearing for Settlement | 对清分明细做账期成熟判断、聚合、锁定和形成结算头寸的过程。 |
| 结算头寸 | Settlement Position | 可被结算单消费的标准待结算资金头寸。 |
| 可结算 | Settlement Eligible | 结算头寸已达到账期并满足结算条件。 |
| 结算批次 | Settlement Batch | 一次运营或系统发起的结算操作批次。 |
| 结算单 | Settlement Bill | 面向一个结算对象生成的结算凭证和入账驱动单。 |
| 账务入账单 | Accounting Posting Order | 清结算平台向账户账务平台发起入账的内部指令。 |
| 可提现余额 | Withdrawable Balance | 账户账务平台维护的商户账户余额事实，不由清结算平台直接维护。 |
| 幂等键 | Idempotent Key | 同一业务请求重复提交时用于返回同一结果的业务键。 |
| 请求指纹 | Request Hash | 同一幂等键下用于校验请求参数是否一致的指纹。 |


# 限界上下文

## 1. 来源事件接入上下文

接收外部业务平台推送的标准履约完成事件，负责幂等、参数指纹校验和入站事件状态管理。

P0 接收事件：

```text
FULFILLMENT_CONFIRMED
```

不接收：退款、撤销、冲正、提现。

## 2. 清分上下文

根据来源事件生成清分结果和清分明细。清分上下文使用标准领域字段，不依赖旧系统表结构。

输出：

```text
ClearingResult
ClearingResultItem
```

## 3. 结算头寸上下文

将清分明细生成结算头寸，并维护在途、可结算、锁定、入账中、入账成功、入账失败、未知等状态。

输出：

```text
SettlementPosition
```

## 4. 结算单上下文

统一处理单条和批量结算确认，生成批次、结算单、结算明细。

核心命令：

```java
SettlementBillNo confirmMerchantSettlement(ConfirmMerchantSettlementCommand command);
```

## 5. 账务入账编排上下文

基于结算单生成账务入账单，调用账户账务平台标准端口，并处理成功、失败、未知状态。

## 6. 标准查询上下文

提供清结算平台标准查询接口，例如结算头寸查询、结算单查询、账务入账单查询。它不承诺页面展示口径。


# 结算头寸状态机

## 1. 状态定义

| 状态 | 说明 | 是否可结算 | 是否终态 |
|---|---|---:|---:|
| `IN_TRANSIT` | 在途，履约已完成但未到可结算时间。 | 否 | 否 |
| `ELIGIBLE` | 已到账期，可被结算。 | 是 | 否 |
| `LOCKED` | 已被结算单锁定，防重复结算。 | 否 | 否 |
| `ACCOUNTING_PROCESSING` | 对应结算单正在账务入账。 | 否 | 否 |
| `ACCOUNTING_FAILED` | 账务入账失败，可随结算单重试。 | 否 | 否 |
| `UNKNOWN` | 账务结果未知，需先查询账务平台。 | 否 | 否 |
| `ACCOUNTED` | 已入账成功。 | 否 | 是 |

## 2. 状态流转矩阵

| 当前状态 | 触发事件 | 前置条件 | 领域动作 | 目标状态 | 失败目标状态 | 是否幂等 | 是否可重试 | 测试用例 |
|---|---|---|---|---|---|---|---|---|
| - | createPosition | 清分明细生成 | 创建结算头寸 | `IN_TRANSIT` | - | 是 | 是 | TC-POS-001 |
| `IN_TRANSIT` | mature | `eligible_time <= now` | 标记可结算 | `ELIGIBLE` | `IN_TRANSIT` | 是 | 是 | TC-POS-002 |
| `ELIGIBLE` | lockForBill | 结算确认命令校验通过 | 写入 `locked_bill_no` | `LOCKED` | `ELIGIBLE` | 是 | 是 | TC-POS-003 |
| `LOCKED` | startAccounting | 结算单进入入账 | 写入 `posting_no` | `ACCOUNTING_PROCESSING` | `LOCKED` | 是 | 是 | TC-POS-004 |
| `ACCOUNTING_PROCESSING` | accountingSuccess | 账务成功 | 标记入账成功时间 | `ACCOUNTED` | `UNKNOWN` | 是 | 是 | TC-POS-005 |
| `ACCOUNTING_PROCESSING` | accountingFailed | 账务明确失败 | 记录失败原因 | `ACCOUNTING_FAILED` | `UNKNOWN` | 是 | 是 | TC-POS-006 |
| `ACCOUNTING_PROCESSING` | accountingUnknown | 调用超时或结果不明 | 记录待查询 | `UNKNOWN` | `UNKNOWN` | 是 | 是 | TC-POS-007 |
| `ACCOUNTING_FAILED` | retryAccounting | 结算单重试 | 跟随结算单重新入账 | `ACCOUNTING_PROCESSING` | `ACCOUNTING_FAILED` | 是 | 是 | TC-POS-008 |
| `UNKNOWN` | queryAccountingSuccess | 查询到账务成功 | 回正成功 | `ACCOUNTED` | `UNKNOWN` | 是 | 是 | TC-POS-009 |
| `UNKNOWN` | queryAccountingFailed | 查询到账务失败 | 回正失败 | `ACCOUNTING_FAILED` | `UNKNOWN` | 是 | 是 | TC-POS-010 |
| `LOCKED` | cancelBillBeforeAccounting | 结算单未入账前取消 | 释放锁定 | `ELIGIBLE` | `LOCKED` | 是 | 否 | TC-POS-011 |

## 3. 禁止流转

- `ACCOUNTED` 不能回到任何非终态。
- `ACCOUNTING_PROCESSING` 不能被运营直接取消。
- `UNKNOWN` 不能直接重复入账，必须先查账务平台。
- `ELIGIBLE` 之外的头寸不能被新结算单锁定。


# 结算单状态机

## 1. 状态定义

| 状态 | 说明 | 是否可取消 | 是否可重试 | 是否终态 |
|---|---|---:|---:|---:|
| `CREATED` | 结算单已创建。 | 是 | 否 | 否 |
| `CONFIRMED` | 结算确认完成，准备入账。 | 是 | 是 | 否 |
| `ACCOUNTING_PROCESSING` | 账务入账中。 | 否 | 否 | 否 |
| `ACCOUNTED` | 入账成功。 | 否 | 否 | 是 |
| `ACCOUNTING_FAILED` | 入账失败。 | 否 | 是 | 否 |
| `UNKNOWN` | 账务结果未知。 | 否 | 需先查询 | 否 |
| `CANCELED` | 入账前取消。 | 否 | 否 | 是 |

## 2. 状态流转矩阵

| 当前状态 | 触发事件 | 前置条件 | 领域动作 | 目标状态 | 失败目标状态 | 是否幂等 | 是否可重试 | 测试用例 |
|---|---|---|---|---|---|---|---|---|
| - | createBill | 头寸锁定成功 | 创建结算单 | `CREATED` | - | 是 | 是 | TC-BILL-001 |
| `CREATED` | confirm | 操作人确认 | 记录确认信息 | `CONFIRMED` | `CREATED` | 是 | 是 | TC-BILL-002 |
| `CONFIRMED` | startAccounting | 账务入账单创建 | 调用账务平台 | `ACCOUNTING_PROCESSING` | `ACCOUNTING_FAILED` | 是 | 是 | TC-BILL-003 |
| `ACCOUNTING_PROCESSING` | accountingSuccess | 账务明确成功 | 回写账务编号 | `ACCOUNTED` | `UNKNOWN` | 是 | 是 | TC-BILL-004 |
| `ACCOUNTING_PROCESSING` | accountingFailed | 账务明确失败 | 记录失败原因 | `ACCOUNTING_FAILED` | `UNKNOWN` | 是 | 是 | TC-BILL-005 |
| `ACCOUNTING_PROCESSING` | accountingUnknown | 超时或结果不明 | 标记未知 | `UNKNOWN` | `UNKNOWN` | 是 | 是 | TC-BILL-006 |
| `ACCOUNTING_FAILED` | retryAccounting | 使用同一账务幂等键 | 再次调用账务 | `ACCOUNTING_PROCESSING` | `ACCOUNTING_FAILED` | 是 | 是 | TC-BILL-007 |
| `UNKNOWN` | queryAccountingSuccess | 查询到账务成功 | 回正成功 | `ACCOUNTED` | `UNKNOWN` | 是 | 是 | TC-BILL-008 |
| `UNKNOWN` | queryAccountingFailed | 查询到账务失败 | 回正失败 | `ACCOUNTING_FAILED` | `UNKNOWN` | 是 | 是 | TC-BILL-009 |
| `CREATED` | cancel | 未发起账务 | 释放头寸 | `CANCELED` | `CREATED` | 是 | 否 | TC-BILL-010 |
| `CONFIRMED` | cancelBeforeAccounting | 未发起账务 | 释放头寸 | `CANCELED` | `CONFIRMED` | 是 | 否 | TC-BILL-011 |

## 3. 幂等行为

重复确认结算时：

| 场景 | 处理 |
|---|---|
| `idempotent_key` 相同且 `request_hash` 相同 | 返回原 `bill_no`。 |
| `idempotent_key` 相同但 `request_hash` 不同 | 拒绝，错误码 `CCS_IDEMPOTENT_CONFLICT`。 |
| 头寸已被其他结算单锁定 | 拒绝，错误码 `CCS_POSITION_LOCKED`。 |


# 账务平台对接契约

## 1. 设计原则

清结算平台不绑定当前任何已有 facade 方法名。P0 需要账户账务平台提供标准端口：

```java
SettlementAccountingPostResult postSettlement(SettlementAccountingPostCommand command);
SettlementAccountingPostResult querySettlementPosting(String accountingRequestNo);
```

如果账户账务平台已有旧接口，可由账户账务平台或适配器实现该标准端口，不反向修改清结算平台领域模型。

## 2. SettlementAccountingPostCommand

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `callerSystem` | String | 是 | `CLEARING_SETTLEMENT_PLATFORM`。 |
| `bizDomain` | String | 是 | 业务域。 |
| `bizNo` | String | 是 | 结算单号 `bill_no`。 |
| `accountingScene` | String | 是 | `MERCHANT_SETTLEMENT_POSTING`。 |
| `settlementTargetType` | String | 是 | P0 为 `MERCHANT`。 |
| `settlementTargetNo` | String | 是 | 商户编号。 |
| `postingAmountCent` | Long | 是 | 入账金额，单位分。 |
| `idempotentKey` | String | 是 | 账务幂等键。 |
| `requestHash` | String | 是 | 账务请求指纹。 |
| `billNo` | String | 是 | 清结算结算单号。 |
| `batchNo` | String | 是 | 清结算批次号。 |

## 3. SettlementAccountingPostResult

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `postingStatus` | String | 是 | `SUCCESS/FAILED/UNKNOWN`。 |
| `accountingRequestNo` | String | 成功或已受理时必填 | 账户账务请求号。 |
| `fundAccountFlowNo` | String | 成功时必填 | 账户流水号或等价可追踪流水引用。 |
| `failCode` | String | 失败时必填 | 失败码。 |
| `failMessage` | String | 失败时必填 | 失败原因。 |

## 4. 回写要求

清结算平台必须将以下字段回写到 `ccs_accounting_posting_order` 和 `ccs_settlement_bill`：

```text
accounting_request_no
fund_account_flow_no
posting_status
posting_success_time
fail_reason
```

## 5. UNKNOWN 处理

当调用超时、网络异常或账户账务平台返回结果不明时，清结算平台进入 `UNKNOWN`，只能通过 `querySettlementPosting` 查询结果，不允许直接再次入账。
