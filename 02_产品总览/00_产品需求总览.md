# 产品需求总览

## 1. 一句话目标

建设本地生活美食业务的封板/下架产品能力：当商品或店铺因运营、合规、履约或商户状态变化需要停止售卖时，系统应支持后台执行下架/封板，用户端不再进入新购买链路，同时保留历史订单、交易快照、售后与结算记录。

## 2. 版本范围

| 类型 | 内容 |
| --- | --- |
| 本期纳入 | 商品单个下架、商品批量下架、商品上架、商品状态筛选、用户端已下架提示、历史订单保留、售后/结算影响边界。 |
| 本期待确认 | 店铺封板后台入口、店铺恢复规则、未消费订单的准确范围、待付款/待核销订单处理、下架后是否冻结结算/提现。 |
| 本期不纳入 | 商户入驻、商品新建全流程、优惠券完整能力、结算平台化、提现、分账、技术方案、测试用例。 |

## 3. 角色与核心诉求

| 角色 | 核心诉求 |
| --- | --- |
| 平台运营 | 能下架违规、异常或需停止售卖的商品/店铺，并看到成功/失败原因。 |
| 商户/供应商 | 能查看自身商品状态，必要时维护商品信息或恢复售卖。是否允许自主操作需确认。 |
| C 端用户 | 不再购买已下架商品/店铺的新单，但能查看历史订单和售后。 |
| 财务/运营 | 下架/封板不能破坏已产生订单的结算、退款、提现等历史链路。 |

## 4. 主流程

```text
选择商品或店铺
  -> 系统校验对象状态、权限、未消费订单
  -> 允许下架/封板
  -> 商品/店铺进入不可新购状态
  -> 用户端隐藏购买入口或展示已下架页
  -> 历史订单、交易快照、售后、结算记录继续可查
```

异常流程：

```text
选择商品或店铺
  -> 系统发现存在未消费订单或状态不允许
  -> 单个操作失败，展示原因
  -> 批量操作过滤不可处理对象，只处理可下架对象
```

## 6. 产品闭环

本功能的闭环不是“把商品状态改成已下架”这么简单，而是同时满足：

1. 后台能发起操作。
2. 系统能校验状态和未消费订单。
3. 操作成功/失败有明确反馈。
4. 用户端新购买入口被收口。
5. 历史订单、交易快照、售后和结算记录保留。
6. 可恢复场景有明确上架或审核规则。
7. 所有未明确的业务口径进入待确认清单。
