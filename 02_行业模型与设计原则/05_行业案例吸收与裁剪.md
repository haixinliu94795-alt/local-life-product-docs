# 行业案例吸收与裁剪

## 1. 本章结论

个人博主文章中的案例不是本地生活/电商大厂技术架构，但它提供了一批结算系统通用业务能力。V010 的处理原则是：

```text
吸收通用结算能力；不吸收传统行业流程边界；不扩大 P0 实现范围。
```

## 2. 吸收能力清单

| 文章能力点 | 是否吸收 | 我们的落点 | P0 实现 |
|---|---:|---|---:|
| 结算数据源 | 是 | `SourceEvent` 标准业务事实 | 是 |
| 结算对象信息来源 | 是 | 外部商户/主体系统通过 Adapter 提供主体信息 | 是，最小化 |
| 结算单设计 | 是 | `SettlementBill / SettlementBillItem` | 是 |
| 结算模式：银行卡/虚拟户 | 是 | `settlement_mode` | P0 只 `INTERNAL_ACCOUNT` |
| 结算周期：T/D/S/周/月 | 是 | `settlement_cycle_type` | P0 核销后 T+N |
| 结算发起方：自动/自助 | 是 | `initiation_mode` | P0 只后台手动 |
| 最小/最大结算金额 | 是 | 策略版本字段 | P0 默认关闭 |
| 留存金额 | 是 | 策略版本字段 + 结算单字段 | P0 默认 0 |
| 规则绑定对象 | 是 | `ccs_settlement_policy_binding` | 是，支持通用和商户级绑定 |
| 客户账单 | 是 | `SettlementBill` 查询投影 | P0 提供标准查询，不做 BFF 展示 |
| 对账意识 | 是 | 数据血缘和对账埋点 | P0 只埋点，不做完整工作台 |
| 发票/税票 | 不进 P0 | 后续 Financial Document Context | 否 |
| 收款/付款单 | 不进 P0 | 出款/收款平台边界 | 否 |
| 采购、验收、合同 | 不进本平台 | 上游业务域 | 否 |

## 3. 行业案例到本平台模型映射

| 行业表达 | 本平台领域语言 | 说明 |
|---|---|---|
| 结算数据源 | `SourceEvent` | 来源业务事实，不承载清分结果。 |
| 清算到商户账户 | `SettlementPosition` | 表达一笔待结算头寸生命周期。 |
| 结算单 | `SettlementBill` | 结算确认后形成的主单。 |
| 结算到账户 | `AccountingPostingOrder` | 编排账户账务平台入账。 |
| 虚拟户 | 账户账务平台资金账户 | 清结算平台不维护余额事实。 |
| 自助/自动结算 | `initiation_mode` | P1/P2 能力，P0 只运营手动。 |
| 最小结算金额 | `min_settlement_amount_cent` | P0 默认 0，未来自动结算使用。 |
| 留存金额 | `reserve_policy_type / reserve_amount_cent / reserve_ratio_bp` | P0 不启用，但账单可承载。 |
| 规则配置 | `ClearingSettlementPolicy` | 不叫普通配置表，而是策略上下文。 |

## 4. 为什么不吸收发票、收款、付款

发票、收款、付款在传统行业案例中常和结算系统耦合，但在本地生活/电商平台中应拆分为独立平台：

- 清结算平台负责清分、清算、结算单、账务入账编排。
- 账户账务平台负责余额、流水、账务请求。
- 出款/提现平台负责银行卡出款或提现。
- 发票/税务平台负责发票、税务、税务归档。
- 收款能力由支付/收款平台承载。

把这些能力都并入清结算平台会破坏平台边界，不符合大规模平台化治理。

## 5. 开发约束

P0 研发只能实现表内默认开启的能力：`INTERNAL_ACCOUNT`、`OPERATOR_MANUAL`、核销后 T+N。其他枚举和字段属于平台长期建模，不得在 P0 中引入额外流程。
