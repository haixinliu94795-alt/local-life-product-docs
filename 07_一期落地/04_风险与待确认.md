# 风险与待确认

## 1. 风险

| 风险 | 等级 | 处理 |
|---|---|---|
| 清结算平台退化成普通结算记录表 | P0 | 强制按清分/清算/结算三层建模。 |
| 旧 finance_detail 和新待结算池双写不一致 | P0 | 灰度期双读对比，对账发现差异。 |
| 入账成功但本地状态失败 | P0 | UNKNOWN 状态 + 幂等查询补偿。 |
| 重复结算/重复入账 | P0 | 唯一索引 + 幂等键 + 状态机。 |
| 金额口径不一致 | P1 | 清分结果、规则快照、服务端重算。 |
| 商户端已结算与可提现余额理解不一致 | P1 | 明确展示口径，余额以账户账务为准。 |
| 历史数据迁移风险 | P1 | 一期不全量迁移，旧表只读兼容。 |
| 规则过度设计 | P2 | 一期不用 DSL，只做版本和快照。 |

## 2. 待确认

| 编号 | 问题 | 默认假设 |
|---|---|---|
| Q-001 | 结算凭证是否必须上传？ | 按原型要求必须上传，最多 5 张。 |
| Q-002 | 批量结算是否允许跨业务类型？ | 不允许，同一商户 + 同一业务/结算场景。 |
| Q-003 | 结算后退款一期是否自动冲正？ | 不做自动冲正，P2 做逆向清分和冲正。 |
| Q-004 | 是否需要双人审核？ | 一期不做双审，但状态机和操作日志预留。 |
| Q-005 | 历史 local_settlement_record 是否迁移？ | 一期不全量迁移，只做查询兼容。 |
| Q-006 | 团餐是否并入一期？ | 默认 P1，除非业务要求提前。 |
