# V009 本版变更说明

## 1. 本章结论

V009 不是重做清结算平台主模型，而是在 V008 的基础上吸收行业案例中适合电商/本地生活平台的成熟能力，重点加强清分结算策略、结算模式、发起方式、金额阈值、留存策略和规则绑定。

## 2. 为什么吸收这些能力

个人博主文章覆盖了多类结算系统案例，其中有些属于传统行业，如发票、收款、采购验收、政务付款等，不适合进入我们 P0 主设计；但其中关于结算周期、结算模式、规则绑定、最小/最大结算金额、留存金额、账单和对账的内容，属于清结算平台通用能力，适合沉淀到我们的策略模型中。

## 3. 本版新增内容

| 类别 | 新增内容 | 落点 |
|---|---|---|
| 行业案例裁剪 | 新增行业案例吸收与裁剪文档 | `02_行业模型与设计原则/05_行业案例吸收与裁剪.md` |
| 策略模型 | 新增结算模式、发起方式、最小/最大金额、留存策略 | `06_策略与清分设计/` |
| DDL | 新增 `ccs_settlement_policy_binding`；策略版本新增模式/发起/阈值/留存字段 | `08_数据模型与存储设计/03_DDL_V009_P0.sql` |
| 接口 | PolicyCommand 增加策略字段；新增策略绑定命令 | `09_接口契约与事件协议/openapi.yaml` |
| ADR | 增加行业案例吸收、策略绑定、结算模式裁剪决策 | `15_架构决策记录_ADR/` |
| 测试 | 增加策略匹配、规则绑定、最小金额、留存默认值测试 | `13_测试验收/` |

## 4. 保持不变的边界

- 不把旧系统表反向定义为清结算平台模型。
- 不让 BFF 页面展示口径定义底层状态。
- 不把退款、出款、发票、收款、付款、税务并入 P0。
- 不新增冻结/止付状态。
- 不改变 `SourceEvent -> ClearingResult -> SettlementPosition -> SettlementBill -> AccountingPostingOrder` 主模型。
