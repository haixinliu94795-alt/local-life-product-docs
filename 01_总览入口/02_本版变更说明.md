# V010 本版变更说明

## 1. 变更背景

V009 已完成行业案例吸收和策略模型增强，但评审发现 P0 正确性闭环仍有若干空白：账期成熟任务未规格化、批量结算无单次边界、UNKNOWN 补偿缺 SLA、操作日志事务策略不清、request_hash 规则未标准化、导出接口存在大数据量风险。

本版只修正这些会影响开发安全和资金正确性的内容，不引入容量规划和大数据量优化。

## 2. 变更清单

| 编号 | 变更项 | 变更结果 |
|---|---|---|
| V010-01 | 账期成熟任务 | 新增 `SettlementPositionMaturityJob` 最小规格，5 分钟一次，单轮 200 条，幂等推进。 |
| V010-02 | 批量结算上限 | `ConfirmSettlementCommand.positionNos` 最大 500 条，超限返回 `CCS_BATCH_LIMIT_EXCEEDED`。 |
| V010-03 | UNKNOWN 补偿 SLA | 自动核查 2 分钟一次，5 次或 30 分钟未回正进入人工诊断。 |
| V010-04 | 操作日志事务策略 | 核心事务 afterCommit 后写日志，日志失败不回滚结算主事务。 |
| V010-05 | 导出能力边界 | P0 删除 `/eligible/export` 定位，后台只提供分页查询。 |
| V010-06 | request_hash 标准化 | 统一 SHA-256、Canonical JSON、字段排序、数组排序、字段集合。 |
| V010-07 | 性能容量边界 | 容量规划、分片、百万导出、归档、压测全部标记为非 P0。 |

## 3. 未改变的设计

- 主模型仍为 `SourceEvent -> ClearingResult -> ClearingResultItem -> SettlementPosition -> SettlementBill -> AccountingPostingOrder`。
- 结算配置仍领域化为 `ClearingSettlementPolicy`。
- P0 不做退款、冻结、出款、BFF 口径适配。
- 旧系统不反向定义底层模型。
