# 清结算平台 V008 正式开发方案总览

## 本章结论

V008 是一版面向设计、开发、测试阶段的高质量 SDD 技术方案。它不新增上线部署内容，也不围绕旧系统修补，而是按照电商/本地生活成熟清结算平台的方式重新组织：先讲行业模型和边界，再讲总体架构、DDD、流程状态机、策略、数据、接口、一致性、扩展性、可观测性、测试和任务卡。

## 架构主线

<img src="../assets/diagrams/01_capability_map.png" width="920" alt="清结算能力地图">

清结算平台的主链路是：

```text
业务履约/核销事实
  -> SourceEvent 标准事件
  -> ClearingResult 清分结果
  -> ClearingResultItem 多参与方金额项
  -> SettlementPosition 结算头寸
  -> SettlementBatch / SettlementBill 结算批次和结算单
  -> AccountingPostingOrder 账务入账单
  -> 账户账务平台入账
```

## P0 实现范围

P0 只实现本地生活普通商品核销完成后的正向清结算闭环：标准事件接入、清分结算策略、清分、头寸、账期成熟、确认结算、账务入账、状态回写、测试验收。

## P0 明确非范围

冻结/止付、退款/冲正、自动出款、完整对账差错、商户端 BFF 展示适配、上线部署/灰度回滚不进入本版。

## 设计质量目标

| 质量目标 | 设计机制 |
|---|---|
| 资金准确性 | 分单位金额、明细汇总校验、账务回写、数据血缘。 |
| 幂等防重 | 业务键、幂等键、request_hash、状态机防重。 |
| 一致性补偿 | 本地事务、外部账务 UNKNOWN 查询、失败重试。 |
| 可追溯性 | 全链路编号贯穿 event、clearing、position、bill、posting、accounting。 |
| 可扩展性 | 多业务线、多参与方、策略快照，不改主链路。 |
| 可测试性 | 状态机矩阵、资金准确性、幂等并发、异常补偿测试。 |
