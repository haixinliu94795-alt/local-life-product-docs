# 结算平台需求提取清单 V001

> 来源：`本地生活_美食封板_V001_20260613` 原型交付包中结算、财务、商家应付、结算配置、分账规则、账户首页、待结算明细、结算详情等页面。本文只做“需求提取”，不直接等同于结算平台完整设计。结算平台完整设计应以行业成熟清结算中心为蓝本，当前需求作为一期纳入范围。

## 2. 需求总览

| 需求域 | 需求结论 |
|---|---|
| 业务接入 | 需要支持普通业务、邀新业务、团餐业务、渠道权益券业务，并为后续新业务线保留扩展能力。 |
| 清分计算 | 需要把订单金额、实付金额、优惠抵扣、会员折扣、金豆、代金券、分佣、平台/商家成本拆分为标准清分结果。 |
| 待结算 | 需要形成待结算明细，结算周期起点为订单核销时间或业务完成时间，加配置账期后进入可结算范围。 |
| 后台结算 | 平台财务可在商家应付/渠道应收列表筛选、导出、单条或批量发起结算。 |
| 结算校验 | 批量结算时必须校验是否选择数据、是否同一商户、是否存在已结算单据。 |
| 结算单 | 确认结算后生成结算记录/结算单，绑定结算明细，变更明细结算状态。 |
| 结算凭证 | 后台结算弹窗要求上传凭证，最多 5 张，支持 jpg/png/jpeg，不能为空。 |
| 结算记录 | 后台需要按结算单号、业务类型、主体、明细范围、结算时间查询结算记录，并展示凭证、账户和操作人。 |
| 规则配置 | 需要配置结算周期、平台/商户成本承担比例、类目维度比例、代金券/优惠券/会员折扣/金豆/推广/邀新成本比例。 |
| 商户资金展示 | 商户端需要展示账户余额、已结算金额、待结算金额、已提现金额、提现中金额、业务统计。 |
| 商户结算查询 | 商户端需要查询账户流水、待结算明细、结算记录、结算详情和订单级应收详情。 |
| 边界能力 | 提现属于出款/提现域；结算平台只需要和账户余额、提现状态做展示和衔接，不把提现本身作为一期结算核心实现。 |

## 3. 功能需求清单

### 3.1 业务接入与资金场景

| 编号 | 需求 | 来源页面 | 一期建议 |
|---|---|---|---|
| REQ-SETTLE-ACCESS-001 | 系统应支持普通商品订单结算，用户支付并完成核销/履约后产生商户应收、平台应收等资金明细。 | `需求`、`商家应付`、`平台分账规则` | 必须实现 |
| REQ-SETTLE-ACCESS-002 | 系统应支持邀新/推广业务订单结算，包含邀新用户分佣、分佣扣税、税后实收、平台/商家分佣成本。 | `需求`、`商家应付_1`、`平台分账规则` | 可作为一期字段兼容，核心链路可二期深化 |
| REQ-SETTLE-ACCESS-003 | 系统应支持团餐业务结算，包含成团价、平台分成比、商户分成、团长分佣、拉新分佣和平台/商家拉新成本。 | `商家应付_2`、`详情-团餐` | 可纳入一期模型，按当前业务优先级决定是否实现 |
| REQ-SETTLE-ACCESS-004 | 系统应支持渠道权益券场景，渠道发放权益券，用户核销/完成后形成平台应收渠道、平台应付商户等应收应付关系。 | `需求`、`结算` | P1/P2，需独立渠道结算边界 |
| REQ-SETTLE-ACCESS-005 | 系统应支持代金券订单、普通订单、代金券抵扣订单三类订单类型，并按不同公式计算应付商家和平台应收。 | `商家应付`、`商家应付_1`、`平台分账规则` | 一期至少在清分模型预留 |

### 3.2 清分与金额计算

| 编号 | 需求 | 来源页面 | 一期建议 |
|---|---|---|---|
| REQ-CLEARING-001 | 订单金额应等于订单实付、平台优惠券抵扣、商家优惠券抵扣、会员折扣、金豆抵扣等项目之和。 | `平台分账规则` | 必须实现清分口径 |
| REQ-CLEARING-002 | 订单实付支持正向计算和逆向校验：正向为订单金额减优惠抵扣，逆向为应付商家金额、平台运营账户、用户分佣金额之和。 | `平台分账规则` | 一期至少做金额校验 |
| REQ-CLEARING-003 | 普通/邀新订单应付商家金额按订单金额、商品类目占比、分佣商家成本、平台优惠券商家成本、商家优惠券、会员折扣商家成本、金豆商家成本等计算。 | `商家应付`、`平台分账规则` | 必须实现或映射现有计算结果 |
| REQ-CLEARING-004 | 普通/邀新订单平台运营账户应收按订单金额、平台占比、分佣平台成本、优惠券平台成本、会员折扣平台成本、金豆平台成本等计算。 | `商家应付`、`平台分账规则` | 必须实现或映射现有计算结果 |
| REQ-CLEARING-005 | 代金券抵扣订单应付商家金额按实付金额乘商品/商户占比，再扣减邀新和分销商家成本。 | `商家应付`、`详情-普通` | 预留，按业务优先级实现 |
| REQ-CLEARING-006 | 代金券订单只涉及订单金额、实付金额、应付商家金额、平台运营应收，其余优惠/分佣字段为 0。 | `商家应付` | 预留 |
| REQ-CLEARING-007 | 团餐订单应付商家金额按成团价乘商户分成占比，再扣减商家拉新成本。 | `商家应付_2`、`详情-团餐` | 如一期接团餐则必须实现 |
| REQ-CLEARING-008 | 团餐订单平台运营账户应收按成团价乘平台分成占比，支持按团长订单量取阶梯占比，并扣减平台拉新成本。 | `商家应付_2` | P1 |
| REQ-CLEARING-009 | 分佣金额需支持税前金额、扣税金额、税后实收金额，原型中个人所得税按 3% 口径展示。 | `平台分账规则` | P1，需财务确认税务口径 |
| REQ-CLEARING-010 | 订单级详情页需要展示应收金额计算公式，并按订单类型显示不同提示。 | `详情-普通`、`详情-团餐` | 商户端查询一期可支持 |

### 3.3 结算配置

| 编号 | 需求 | 来源页面 | 一期建议 |
|---|---|---|---|
| REQ-RULE-001 | 系统应支持结算周期配置，结算起始时间为订单核销时间或业务完成时间加结算周期天数。 | `结算配置` | 必须实现 |
| REQ-RULE-002 | 系统应支持平台/商户成本承担比例配置，默认平台 100%、商户 0%。 | `结算配置` | 必须实现 |
| REQ-RULE-003 | 输入平台占比后，系统自动计算商户占比；商户占比不可手动修改，商户占比=100%-平台占比。 | `结算配置` | 必须实现 |
| REQ-RULE-004 | 比例值保留一位小数。 | `结算配置` | 必须实现 |
| REQ-RULE-005 | 配置项支持统一配置和类目配置，类目配置按“一级/二级/三级类目”展示。 | `结算配置` | P1，模型一期预留 |
| REQ-RULE-006 | 结算比例取值优先使用最低层级子类目配置，子类目为空时向上级类目继承。 | `结算配置` | P1，模型一期预留 |
| REQ-RULE-007 | 配置项应覆盖会员折扣、平台优惠券、金豆抵扣、推广分佣、邀新分佣、代金券结算比例。 | `结算配置` | 模型必须支持，一期按当前用到项实现 |

## 4. 平台级补充要求：不是原型显式需求，但属于成熟结算平台必备

| 编号 | 要求 | 纳入理由 | 实施阶段 |
|---|---|---|---|
| NFR-SETTLE-001 | 结算单必须有状态机，至少覆盖待确认、处理中、成功、失败、取消、未知。 | 防止“记录已生成但账务未入账”或“账务成功但本地未知”。 | 一期必须 |
| NFR-SETTLE-002 | 结算单、结算明细、账务入账请求必须具备幂等键和唯一约束。 | 防止重复点击、重复结算、重复入账。 | 一期必须 |
| NFR-SETTLE-003 | 清分结果、待结算项、结算单、账务流水之间必须可追溯。 | 支撑商户查询、后台审计和财务核查。 | 一期必须 |
| NFR-SETTLE-004 | 金额计算使用定点小数，金额汇总必须有一致性校验。 | 防止浮点误差和账单不平。 | 一期必须 |
| NFR-SETTLE-005 | 结算平台只生成结算单并调用账户账务入账；提现/出款属于出款平台边界。 | 保持领域边界清晰。 | 一期必须 |
| NFR-SETTLE-006 | 对账中心应覆盖订单-清分、清分-待结算、待结算-结算单、结算单-账务流水。 | 防漏结、错结、重结。 | P1 |
| NFR-SETTLE-007 | 规则必须有版本和快照，历史订单按当时规则结算。 | 防止规则修改影响历史结算。 | 一期模型预留，P1 强化 |
| NFR-SETTLE-008 | 所有后台操作必须记录操作人、操作时间、操作对象、前后状态、失败原因。 | 支撑审计和问题追溯。 | 一期必须 |

## 5. 一期实现建议

一期不追求一次性实现全部成熟结算平台能力，但模型必须能容纳完整平台设计。建议一期只做：

1. 普通商品订单接入；
2. 清分结果标准化落表，或至少将现有清分快照映射为标准清分结果；
3. 待结算池；
4. 后台商家应付列表和结算操作；
5. 结算单、结算明细、结算状态机；
6. 结算幂等和账务入账编排；
7. 商户端待结算、结算记录、结算详情；
8. 与账户账务平台和提现平台的边界打通；
9. 操作日志和最小审计。

二期/三期再做：团餐深度规则、邀新/推广分佣完整税务口径、渠道应收、凭证复杂管理、自动结算、自动出款、对账中心、差错处理、规则中心版本治理和跨业务线接入。
