5. 业务链路现状
链路：支付成功 → 清分 → 待结算 → 结算单 → 入账 → 出款 → 对账

链路步骤	状态	证据
1. 支付成功后是否有清分/资金归属计算	部分实现	LocaOrderServiceImpl.addFinance() 在支付回调后创建 local_order_finance_detail 快照（含 payable_to_merchant、platform_receivable 等比例和金额），但不是独立清分平台，而是嵌入订单模块

4. 是否有结算周期/账期处理	部分实现	LocalSettlementConfiguration.billingCycle 字段存在，getWriteOffTime() 用于筛选可结算订单（核销后 N 天才可结算），但无自动触发，全靠运营手动

7. 与账户账务平台的关系
recordProductSettlement 实现：SettlementAccountingApplicationService.recordProductSettlement() 存在且已实现，构建 AccountingRequestCommand（含 batchNo、merchantIdentityId、grossAmount、platformAmount），调用 accountingApplicationExecutor.execute()，写 fund_account 余额和 fund_account_flow 流水。
真正的调用方：LocalSettlementRecordServiceImpl.addSettlementOrder() / addSettlementGmOrder() / addSettlementTsMainOrder() 在后台运营触发结算时调用，callerSystem=SETTLEMENT_PLATFORM，bizDomain=SETTLEMENT。
bizNo / idempotentKey：bizNo = record.statementNumber（结算单号），idempotentKey = "PRODUCT_SETTLEMENT:{statementNumber}:CONFIRM" 或 "SERVICE_ORDER_SETTLEMENT:{statementNumber}:CONFIRM"。
结算入账流水查询：通过 AccountingQueryFacade.pageAccountStatementItems() 查 fund_account_flow，在 flowList() 接口中展示给商户端，category=SETTLEMENT。
是否存在重复计算余额：存在轻微双轨问题——totalSettledAmount（累计结算）直接 SUM local_settlement_record.amount_payable，而真实可用余额从 fund_account.available_amount 取，两者口径不同（前者是毛结算金额，后者是净入账余额）。不是错误，但会让商户看到"已结算"和"余额"对不上的疑问。
