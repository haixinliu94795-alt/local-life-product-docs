# 清结算平台 V003 开发准入总览

## 1. 本版定位

V003 是清结算平台一期普通商品落地准入版，目标是让研发/Codex 可以基于文档直接进入实现，不再停留在领域蓝图层。

本版只做本地生活普通商品核销完成后的清结算入账闭环，不做冻结/止付、自动出款、团餐、邀新、渠道和完整对账工作台。

## 2. 核心资金主线

<img src="../09_资产/diagrams/main_chain.png" width="860" alt="P0普通商品清结算主链路">

```text
支付成功形成在途资金事实
  -> 核销完成
  -> 清分生成订单级资金归属
  -> 清算生成待结算项并判断账期成熟
  -> 后台商家应付只展示可结算数据
  -> 运营确认单条/批量结算
  -> 生成结算批次、结算单、结算明细
  -> 调用账户账务平台入账
  -> 商户可提现余额增加
```

## 3. 删除冻结/止付

P0 不包含冻结/止付能力，也不建立对应状态和字段。退款、取消核销通过最小规则处理：结算前作废或调整，结算中等待结算终态，结算后生成负向调整。

## 4. 三套状态机

<img src="../09_资产/diagrams/state_overview.png" width="860" alt="三套状态机联动">

### 待结算项状态

```text
IN_TRANSIT -> ELIGIBLE -> LOCKED -> ACCOUNTING_PROCESSING -> ACCOUNTED
                                \-> ACCOUNTING_FAILED -> ACCOUNTING_PROCESSING
IN_TRANSIT/ELIGIBLE -> CANCELED
```

### 结算单状态

```text
CREATED -> CONFIRMED -> ACCOUNTING_PROCESSING -> ACCOUNTED
                                      \-> ACCOUNTING_FAILED -> ACCOUNTING_PROCESSING
                                      \-> UNKNOWN -> ACCOUNTED/ACCOUNTING_FAILED
CREATED/CONFIRMED -> CANCELED
```

### 账务入账单状态

```text
INIT -> REQUESTING -> SUCCESS
                  \-> FAILED -> REQUESTING
                  \-> UNKNOWN -> SUCCESS/FAILED/REQUESTING
```

## 5. P0 数据模型

P0 表清单：

| 表名 | 职责 |

| `ccs_pending_settlement_item` | 待结算项生命周期。 |

完整 DDL 见：`03_数据设计/02_DDL_V003_P0.sql`。

## 6. 单条和批量结算

单条和批量统一走：

```java
SettlementBillNo confirmMerchantSettlement(ConfirmMerchantSettlementCommand command);
```

- 单条：`pendingItemNos.size() == 1`。
- 批量：`pendingItemNos.size() > 1`。
- P0 批量全成功/全失败，不做部分成功。

## 7. 账务对接

P0 普通商品先复用现有：

```java
AccountingFacade.recordProductSettlement(...)
```

账务幂等键：

```text
ACCOUNTING:{billNo}:PRODUCT_SETTLEMENT
```

`ACCOUNTED` 成功后，商户端才展示为已结算。

## 8. 退款最小规则

| 场景 | P0 处理 |
|---|---|
| 结算前退款 | 全额转 CANCELED；部分调整 settlement_amount。 |
| 结算中退款 | 退款事件 WAITING，等待结算终态。 |
| 结算后退款 | 生成负向清分和负向待结算项，下一期抵扣。 |

## 9. 新老双轨

- cutover 前旧数据继续旧链路收尾。
- cutover 后新核销订单进入新清结算平台。
- 商户端待结算和已结算在双轨期合并展示。
- 后台旧结算入口必须过滤 cutover 后数据，防重复结算。
