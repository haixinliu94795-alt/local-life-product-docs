# Codex 开发任务总指令

任务名称：清结算平台 P0 普通商品落地实现

## 背景

当前系统没有独立清结算平台。现有结算能力分散在后台 CRUD、账户账务平台、渠道结算和商户 App BFF 中，有真实入账链路，但缺少清分平台、结算状态机、自动触发链路和对账机制。

本次开发不继续修补 `local_settlement_record`，而是新增清结算平台 P0 模块，先完成普通商品核销后结算入账闭环。

## 目标

实现：

```text
核销完成事件
  -> 清分结果
  -> 待结算项
  -> 到账期可结算
  -> 后台商家应付
  -> 单条/批量统一结算
  -> 结算批次/结算单/明细
  -> 账务入账
  -> 商户端待结算/已结算/详情查询
```

## 严格边界

1. P0 不实现冻结/止付。
2. P0 不实现自动出款。
3. P0 不实现团餐、邀新、渠道全链路。
4. P0 不把 `local_settlement_record` 当新平台主表。
5. P0 普通商品入账复用 `AccountingFacade.recordProductSettlement`。
6. P0 单条结算和批量结算必须共用一个核心方法。
7. P0 必须实现三套状态机：待结算项、结算单、账务入账单。
8. P0 必须按 DDL 创建新表，不允许只在旧表加字段冒充平台化。

## 开发顺序

1. 新建模块/包和 DDL。
2. 实现来源事件入库和幂等。
3. 实现普通商品清分快照映射。
4. 实现待结算项生成和到账期扫描。
5. 实现后台商家应付查询。
6. 实现统一确认结算方法。
7. 实现账务入账编排。
8. 实现账务失败/未知/重试。
9. 实现商户端 BFF 查询。
10. 实现新老双轨数据合并。
11. 补齐测试。

## 输出要求

每个任务提交时必须说明：

- 修改文件；
- 新增表/字段；
- 状态机转移点；
- 幂等键；
- 测试命令和结果；
- 未覆盖事项。

最后必须输出：

```text
本次修改范围
已执行测试
未执行测试及原因
风险和回滚方式
```
