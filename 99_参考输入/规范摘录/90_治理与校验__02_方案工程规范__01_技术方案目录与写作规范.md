# 技术方案目录与写作规范

## 1. 目标

技术方案应服务于架构评审、开发落地、测试验收、上线发布和长期维护。方案不是聊天记录，不是过程报告，也不是一次性 Codex 输入包。

## 2. 当前有效方案目录

当前有效方案集中在：

```text
00_正式技术方案/
01_平台化建设总纲/
02_账户账务平台/
03_资金出款编排平台/
04_支付平台/
05_收款核心/
06_出款核心/
07_支付通道网关/
08_通道账户管理/
09_商家平台/
10_券中心业务员端/
11_本地生活用户端/
12_平台管理后台/
13_跨平台接口_事件_状态_数据表索引/
```

## 3. 平台方案建议章节

平台目录可逐步统一为：

```text
00_总览入口/
01_业务场景与需求/
02_领域模型/
03_流程与状态机/
04_接口契约/
05_数据模型/
06_一致性_幂等_异常_补偿/
07_测试验收/
08_代码落地任务包/
09_上线与运维/
```

已有平台不要求一次性重排。V2.37~V2.46 已完成 `01~13` 顶层平台目录模板第一轮对齐。后续新增方案应优先进入标准目录，不得恢复旧 active 目录或历史生成物。

## 4. 写作要求

每个核心方案至少应说明：

- 业务目标和不做事项；
- 平台职责边界；
- 主关联编号和上下游关联；
- 状态机和非理想路径；
- 接口契约和事件契约；
- 数据模型和字段语义；
- 幂等、一致性、异常、补偿；
- 观测、后台诊断和上线验收；
- Codex 开发落地清单；
- 测试覆盖矩阵。

## 5. 禁止事项

- 不把历史生成物作为当前方案正文；
- 不把迁移过程报告放在当前平台主路径；
- 不用旧口径解释当前实现；
- 不让同一事实在多处各自成为权威源；
- 不在方案中只写 happy path，不写异常和回正路径。

## 6. 平台目录对齐的操作要求

平台内部目录模板对齐必须按 `09_平台内部目录模板对齐规范.md` 执行。每次对齐只做结构归位和引用同步，不借机修改业务语义。

每批对齐至少输出：

- 迁移映射；
- hash 完整性对账；
- README / 索引 / 决策台账路径同步；
- 自动化结构审计结果。

## V2.47 内容质量复核补充

V2.36~V2.46 已完成项目结构自进化治理和 `01~13` 顶层目录模板对齐。V2.47 起进入平台内部内容质量复核阶段。

后续平台复核不再以目录搬迁为主，而是检查：

```text
1. active 文档是否存在旧口径残留；
2. 当前权威源是否清晰；
3. 测试、Codex 任务、上线验收是否覆盖事故路径；
4. 是否存在局部成功冒充全链路成功；
5. 是否需要新增自动化审计防止回退。
```

每个平台内容质量复核应在对应平台内新增总览入口、权威口径清单、测试补强清单、代码准出清单和上线验收规范，并将自动化审计接入 `tools/`、CI 和 pre-commit。
