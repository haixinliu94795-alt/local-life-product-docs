# 全项目一致性收口与代码整改前准入规范

## 1. 目的

本规范用于 V2.60 之后的阶段切换：项目已经完成结构治理、平台目录模板对齐、平台内部内容质量复核，后续不能再继续无边界地整理文档，而应进入代码一致性核查、测试补齐和 Codex 代码整改。

## 2.1 关键目录边界

```text
01_平台化建设总纲/：全项目平台化总纲，不替代具体平台实现。
04_支付平台/：支付平台总纲，不替代 05/06/07/08 子平台实现。
13_跨平台接口_事件_状态_数据表索引/：跨平台索引和全局约定，不替代平台 canonical 文档。
90_治理与校验/：治理、门禁、Decision Registry 和自动化，不替代业务方案正文。
```

上述目录不得互相替代。

## 2. 总体准入规则

| 准入项 | 要求 |
|---|---|
| 目录结构 | 必须通过 `project_structure_audit.py` 和 `platform_directory_template_audit.py` |
| 平台 canonical 文档 | 每个平台必须存在内容质量复核入口、权威口径清单、测试补强、代码准出和上线验收文档 |
| 权威源边界 | 01 / 04 / 13 / 90 不得互相替代，不得覆盖具体平台 canonical 文档 |
| Decision Registry | 涉及语义裁决必须先有 ACTIVE 决策，再修改方案和代码 |
| 自动化门禁 | 新增的防回退规则必须能进入专项审计或语义回归 |
| Codex 任务 | 必须先生成任务卡，再编码；不得只读总纲或索引直接编码 |

## 3. 代码整改前标准路径

```text
需求 / 问题 / 事故
  -> GPT 识别涉及平台
  -> 查 01 / 04 / 13 / 90 判断边界
  -> 查对应平台 canonical 文档
  -> 如涉及语义变更，先追加 Decision Registry / ADR
  -> 输出 Codex 任务卡
  -> Codex 按平台代码准出清单整改
  -> 补单测 / 契约测试 / 事故路径测试 / E2E 显式准入
  -> 跑专项审计 + docs_spec_ci + contract_content_audit
  -> 人工评审和上线验收
```

## 4. 需要回方案层处理的情况

出现以下任一情况，不能直接让 Codex 改代码：

```text
字段语义不清
主关联编号不清
状态机是否允许流转不清
DDL 与 Entity 是否一致不清
01 / 04 / 13 / 具体平台之间出现权威源冲突
接口入参 / 出参需要新增或删除
事件 payload 需要新增或删除
补偿语义或人工处理边界需要变化
线上事故暴露出测试和门禁缺口
```

处理方式：先更新方案 canonical 文档，再追加或修订 Decision Registry，再补自动化门禁，最后交给 Codex 改代码。

## 5. V2.60 收口结论

V2.60 以后，文档治理的默认重点不再是继续搬目录或补平台入口，而是：

```text
按平台做代码一致性核查
按具体问题补测试和自动化门禁
按 Codex 任务卡推进代码整改
按上线闭环规范做真实链路验收
```


## 8. SQL 与 DDL 准入规则

后续 GPT / Codex / 人工上线验收凡涉及 SQL，必须先读取对应平台 DDL 或统一表 DDL：

1. 不得凭字段名习惯猜测表字段；
2. 若生产表与技术方案不一致，先报告差异，不得继续写基于假设字段的 SQL；
3. `payment_event_outbox` 的 `bizDomain / bizNo` 在 `payload` JSON 中，不是独立表字段；
4. `payment_event_outbox.aggregate_no` 是聚合编号，如出款核心 `PYO...`，不是 03 的 `FPP...`；
5. 上线 SQL 必须使用 `INIT / PUBLISHING / PUBLISHED / FAILED / DEAD` 等技术方案定义的事件状态，不得引入未定义状态。
## 5.5 兼容性边界原则

当前项目进入自进化维护期后，默认不为历史数据、旧代码、旧内部接口、旧枚举、旧事件、旧字段保留兼容。唯一需要谨慎兼容的是已经与前端联调完成的对外接口契约，包括 URL、入参字段、出参字段和前端已依赖的展示字段。

执行规则：

1. 前端已联调接口字段名可以保留，但进入服务层后必须映射为当前正式方案语义；
2. 内部旧字段、旧枚举、旧事件、旧查询兼容若与当前正式方案冲突，应清理或拒绝；
3. 不因历史数据存在而保留写侧、状态推进侧、事件消费侧或内部查询兼容；
4. Codex 任务中如出现“历史兼容”与当前方案冲突，应优先按当前正式方案执行，并向 GPT / 人反馈冲突；
5. `PAYOUT_PLATFORM`、legacy `bizDomain = PAYOUT`、`sourceSystem / sourceBizNo` 等旧内部口径不得继续作为当前有效实现依据。
