# 策略绑定模型

## 1. 本章结论

策略不应该被复制到每一个商户、渠道、门店或代理对象上。成熟平台应采用“通用策略 + 对象绑定 + 特例覆盖”的方式进行规则治理。

## 2. 设计动机

如果每新增一个商户或代理都复制一份完整结算策略，会导致：

- 策略数量爆炸，运营难以维护。
- 修改通用规则时无法统一生效。
- 历史规则难以追溯。
- 特殊商户策略和默认策略混在一起，审计困难。

因此 V009 新增 `PolicyBinding` 模型：业务对象只引用策略，策略本身集中维护。

## 3. 绑定对象

| 绑定对象 | 枚举 | 说明 |
|---|---|---|
| 全局 | `GLOBAL` | 平台默认策略。 |
| 业务线 | `BUSINESS_SCENE` | 如本地生活普通商品、团餐、按摩等。 |
| 类目 | `CATEGORY` | 餐饮、休闲、按摩等类目。 |
| 商户 | `MERCHANT` | 特殊商户策略。 |
| 渠道 | `CHANNEL` | 渠道业务策略。 |
| 活动 | `CAMPAIGN` | 特定营销活动策略。 |

## 4. 匹配顺序

默认匹配顺序：

```text
MERCHANT > CAMPAIGN > CATEGORY > BUSINESS_SCENE > GLOBAL
```

同层级通过 `binding_priority` 控制，数值越小优先级越高。

## 5. 开发落点

- 表：`ccs_settlement_policy_binding`
- 服务：`PolicyMatchDomainService.matchPolicy(context)`
- 输入：业务场景、商户、类目、活动、发生时间
- 输出：`policy_no`、`policy_version_no`、`rule_snapshot_json`
- 使用方：清分服务和结算头寸生成服务

## 6. P0 裁剪

P0 至少支持：

```text
GLOBAL
BUSINESS_SCENE
MERCHANT
```

`CATEGORY`、`CHANNEL`、`CAMPAIGN` 可以先保留字段和枚举，后续接入。
