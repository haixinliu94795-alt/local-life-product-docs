# 最小/最大结算金额与留存金额策略

## 1. 本章结论

最小结算金额、最大结算金额和留存金额是成熟结算平台常见策略能力。V010 将其纳入策略模型和结算单金额模型，但 P0 默认关闭，不增加普通商品一期流程复杂度。

## 2. 策略定义

| 策略 | 字段 | 含义 | P0 默认 |
|---|---|---|---|
| 最小结算金额 | `min_settlement_amount_cent` | 低于阈值时不自动生成结算单，或累计到后续周期。 | 0，不限制 |
| 最大结算金额 | `max_settlement_amount_cent` | 超过阈值时进入复核、告警或人工确认。 | 0，不限制 |
| 留存策略 | `reserve_policy_type` | 是否从结算金额中留存固定金额或比例。 | NONE |
| 固定留存 | `reserve_amount_cent` | 固定留存金额。 | 0 |
| 比例留存 | `reserve_ratio_bp` | 按比例留存，10000=100%。 | 0 |

## 3. 结算单金额口径

`SettlementBill` 固化以下金额：

```text
total_amount_cent      = 结算明细汇总金额
reserve_amount_cent    = 留存金额，P0 默认 0
posting_amount_cent    = 本次应入账金额 = total_amount_cent - reserve_amount_cent
```

P0 默认：

```text
reserve_amount_cent = 0
posting_amount_cent = total_amount_cent
```

## 4. 为什么 P0 默认关闭

普通商品一期的目标是打通正向清结算入账闭环。如果同时启用最小金额、最大金额和留存，会增加批量结算拦截、复核、累计、留存归还等复杂度。V010 采用“模型预留、默认关闭”的方式，既保证长期平台能力，又不扩大 P0 交付范围。

## 5. 测试要求

P0 至少要验证：

- 默认 0 时不拦截结算。
- `reserve_amount_cent=0` 时 `posting_amount_cent=total_amount_cent`。
- 表字段和 DTO 可以承载策略值。
- P0 代码不得因这些字段默认值而分叉出出款、留存归还或复核流程。
