# P0 批量处理与任务调度边界

## 1. 本章结论

P0 只定义保证正确性所需的批量边界和最小任务规格，不做大数据量处理、不做容量规划、不做任务分片。

必须实现的批量能力只有两类：

1. 账期成熟任务：推进 `IN_TRANSIT -> ELIGIBLE`。
2. 结算确认批量：一次确认有限数量头寸，生成结算单。

## 2. SettlementPositionMaturityJob

| 项 | 规格 |
|---|---|
| 默认频率 | 每 5 分钟一次，配置化 |
| 单轮处理数 | 200 条，配置化 |
| 扫描条件 | `lifecycle_status = IN_TRANSIT and eligible_time <= now()` |
| 排序 | `id asc` |
| 事务策略 | 单条或小批量事务 |
| 失败策略 | 单条失败不影响其他头寸，下轮继续 |
| 非 P0 | 分片、分库分表、动态分配、容量评估 |

## 3. 统一结算确认批量边界

| 项 | 规格 |
|---|---|
| 方法 | `confirmMerchantSettlement(command)` |
| 单次最小数量 | 1 个 `positionNo` |
| 单次最大数量 | 500 个 `positionNo` |
| 超限处理 | 拒绝，返回 `CCS_BATCH_LIMIT_EXCEEDED` |
| 成功策略 | 全成功 / 全失败 |
| 部分成功 | P0 不支持 |
| 事务策略 | 一个结算单一个主事务，事务内创建 batch/bill/items/posting order |

## 4. 为什么仍要有上限

这不是容量规划，而是资金写操作的安全边界。结算确认需要锁定头寸、生成结算明细、创建入账单，如果不限制 `positionNos`，一次错误调用可能导致长事务和不必要的事故面扩大。

## 5. 错误码

```text
CCS_BATCH_LIMIT_EXCEEDED
```

错误语义：

```text
单次结算确认头寸数量超过 P0 上限。请缩小选择范围后分批提交。
```

## 6. 不做事项

```text
百万级批处理
拆批任务
异步结算单生成
商户级任务分片
容量规划
压测目标
```
