# 旧系统映射

## 1. 当前现状

当前系统没有独立清结算平台，结算能力分散在后台结算 CRUD、账户账务平台、渠道结算模块和商户 App BFF 中。已有真实入账能力，但缺少独立清分平台、清算模型、结算状态机、自动触发链路和对账机制。

## 2. 旧表到新模型

| 旧表/能力 | 新模型 | 迁移策略 |
|---|---|---|
| local_order_finance_detail | cs_clearing_result + cs_pending_settlement_item | 一期通过防腐层读取并逐步双写。 |
| gm_order_finance_detail | cs_clearing_result + cs_pending_settlement_item | P1 团餐接入时迁移。 |
| ts_order_finance_detail | cs_clearing_result + cs_pending_settlement_item | 服务订单接入时迁移。 |
| local_settlement_record | cs_settlement_bill | 历史查询兼容，不作为新核心。 |
| local_settlement_record_order | cs_settlement_bill_item | 历史查询兼容。 |
| local_settlement_configuration | cs_settlement_rule | 迁移为规则配置。 |
| channel_settle_record | cs_settlement_bill / 渠道应收应付模型 | P1 渠道统一接入。 |
| fund_account/fund_account_flow | 账户账务平台 | 保持边界，只引用流水。 |

## 3. 迁移阶段

1. 新平台建表，不影响旧链路。
2. 防腐层读取旧数据生成清分和待结算读模型。
3. 后台商家应付列表新旧双读对比。
4. 小范围商户走新清结算链路。
5. 商户端查询切新读模型。
6. 旧结算表转只读归档。
