# 审计与操作日志

## 1. 本章结论

操作日志用于运营审计和问题排查，不是资金事实唯一来源。P0 要保证核心资金状态先提交成功，再通过 afterCommit / REQUIRES_NEW / 异步方式写入操作日志。日志写入失败不得回滚结算主事务。

## 2. 日志记录范围

| 操作 | 聚合 | 日志类型 |
|---|---|---|
| 来源事件接入 | SourceEvent | `SOURCE_EVENT_ACCEPTED` |
| 清分成功 | ClearingResult | `CLEARING_CLEARED` |
| 头寸成熟 | SettlementPosition | `POSITION_MATURED` |
| 结算确认 | SettlementBill | `SETTLEMENT_CONFIRMED` |
| 账务入账成功 | AccountingPostingOrder | `ACCOUNTING_SUCCESS` |
| 账务入账失败 | AccountingPostingOrder | `ACCOUNTING_FAILED` |
| 账务 UNKNOWN | AccountingPostingOrder | `ACCOUNTING_UNKNOWN` |
| UNKNOWN 人工介入 | AccountingPostingOrder | `UNKNOWN_MANUAL_REQUIRED` |

## 3. 事务策略

```text
核心资金主事务提交成功
  -> afterCommit
  -> OperationLogWriter.writeRequiresNew
  -> 日志写入失败只记录错误和指标，不回滚主事务
```

## 4. 日志失败处理

| 场景 | 处理 |
|---|---|
| 日志表写入失败 | 打 error log，增加 `ccs_operation_log_write_failed_total`。 |
| 日志重复写入 | 使用 `operation_no` 幂等。 |
| 日志延迟 | 不影响资金状态。 |
| 审计追补 | 后续可通过补偿任务从聚合状态重建审计线索，P0 不实现。 |

## 5. 审计字段

| 字段 | 说明 |
|---|---|
| `operation_no` | 操作日志编号。 |
| `aggregate_type` | 聚合类型。 |
| `aggregate_no` | 聚合编号。 |
| `operation_type` | 操作类型。 |
| `before_status` | 操作前状态。 |
| `after_status` | 操作后状态。 |
| `operator_no` | 操作人编号。 |
| `operation_source` | 操作来源。 |
| `operation_payload_json` | 扩展信息。 |
